Module 1: S3 Security - Bucket Policies for Defense in Depth

Core Pattern: Use bucket policies to enforce security requirements at the bucket level

Bucket Policy Use Cases

1. Enforce HTTPS Transport

  • Deny requests without SSL (aws:SecureTransport condition)

2. Enforce Encryption at Rest

  • Require SSE-KMS with customer managed key
  • Separate key administrators vs operators (least privilege)
‚ö†Ô∏è Gotcha: Without bucket policy, default encryption can be overridden via API headers. With policy = encryption type must match request

3. Restrict Network Access

  • Limit to VPC endpoint only (aws:SourceVpce condition)
‚ö†Ô∏è Gotcha: VPC endpoint exists ‚â† traffic uses it. Must attach route table to redirect S3 traffic through endpoint (otherwise flows via IGW)

Complementary Tools

AWS Config

Purpose: Detect policy violations

Example: s3-bucket-public-read-prohibited rule generates non-compliance alerts

Access Analyzer for S3

Purpose: Identify external access

S3 feature (not IAM Access Analyzer). Surfaces unintended public/cross-account access

üí° Key Insight: Bucket policies = proactive enforcement, Config/Access Analyzer = detective controls

Module 2: Storage Performance - Parallelization for Throughput

Core Pattern: Distribute operations across multiple threads/inodes to leverage AWS storage's distributed architecture

S3 Optimization Techniques

1. Multipart Upload ‚Üí Break large objects into chunks

  • 5GB file ‚Üí 16MB chunks = 313 PUTs vs 1 PUT
  • Trade-off: Speed costs money ($0.005 per 1,000 requests)
  • "Paying a toll to go faster"

2. Threaded Operations ‚Üí Parallel small file uploads

  • Multi-threaded aws s3 sync significantly faster
  • Each thread handles independent file/chunk

3. S3-to-S3 Copies ‚Üí Use native copy methods

  • aws s3api copy-object or aws s3 cp with Transfer Manager
  • Keeps data inside S3 network (no EC2 bandwidth)
  • Faster + multi-threaded + cheaper

EFS Optimization Techniques

1. Parallel Threads and Inodes ‚Üí Multiple simultaneous operations

  • Leverages EFS distributed storage design
  • Drives aggregate throughput and IOPS

2. Sync Strategy ‚Üí Per-file vs per-block

‚ö†Ô∏è Gotcha: Syncing after each block = dramatic performance degradation. Optimal: Sync after each complete file
üí° Key Insight: AWS distributed storage shines with parallelization‚Äîsingle-threaded operations underutilize available performance

Module 3: Data Migration - Phased Cutover with Hybrid Access

Core Pattern: Use DataSync for bulk transfer, File Gateway for hybrid access, then cutover when ready

Migration Phases

Initial Bulk Transfer ‚Üí DataSync

  • Activate agent (target region, source public IP)
  • Create locations (NFS source, S3 destination)
  • Run task for all existing data

Hybrid Access Period ‚Üí File Gateway + Incremental DataSync

  • File Gateway = NFS interface to S3
  • Apps continue using familiar NFS mount
  • DataSync for incremental changes only

Cache Synchronization ‚Üí Manual refresh after external writes

‚ö†Ô∏è Gotcha: DataSync writes invisible to File Gateway until cache refresh
  • File Gateway maintains metadata cache for performance
  • Gateway writes = auto-update cache
  • External writes (DataSync, direct S3) = manual refresh needed

Final Cutover ‚Üí Gateway-only operations

  • Unmount legacy NFS
  • Delete DataSync resources (order: Task ‚Üí Locations ‚Üí Agent)
  • All writes through Gateway ‚Üí auto-sync to S3
  • S3 data now accessible to other AWS services
üí° Key Insight: File Gateway and DataSync operate independently‚Äîunderstanding cache boundaries prevents "the file exists but I can't see it" production issues

Module 4: AWS Backup - Tiered Protection with Compliance Monitoring

Core Pattern: Implement tiered backup strategy with tag-based automation and continuous compliance monitoring

Backup Plan Design

Two-section structure ‚Üí Rules + Resource assignment

  • Backup rules = scheduling, frequency, retention
  • Resource assignment = what resources to protect

Tiered vaults and plans ‚Üí Gold/Silver designation

  • Gold = mission-critical / Silver = standard resources
  • Separate vaults for isolation and organization

Tag-based resource assignment ‚Üí Automated protection

  • Works across ALL AWS Backup-supported resources (EBS, EFS, RDS, DynamoDB, etc.)
  • Dynamically includes/excludes resources without manual plan updates
‚ö†Ô∏è Gotcha: Tags must match exactly‚Äîtypos exclude resources from protection

On-demand backup alternative ‚Üí Manual resource specification

  • Useful for one-off protection or testing
  • Bypasses tag-based automation

Backup Execution

Cron expressions for scheduling ‚Üí Precise control

  • cron(0 * ? * * *) = hourly backups

Windows VSS integration ‚Üí Application-consistent backups

  • Quiesces databases before snapshot (SQL Server)
  • Enables clean restores without recovery procedures
  • Must enable under "Advanced backup settings"

Recovery Patterns

Service-specific restore behavior ‚Üí No overwrites

  • EBS: Restores as new unattached volume (manual attach to instance)
  • EFS: Restores to timestamped subdirectory in existing filesystem
‚ö†Ô∏è Gotcha: AWS Backup never overwrites primary data‚Äîalways creates new resources

Recovery workflow ‚Üí Attach/mount restored resources

  • EBS: Detach old ‚Üí Delete ‚Üí Restore ‚Üí Attach new ‚Üí Mount
  • EFS: Automatic restore to aws-backup-[timestamp]/ folder

Compliance Monitoring

Audit Manager frameworks ‚Üí Control collections

  • Pre-built controls (frequency, retention, encryption)
  • Custom thresholds (hours vs days, months vs weeks)
  • AWS Config rules under the hood

Non-compliance remediation ‚Üí Re-evaluate after fixes

  • Manual re-evaluation triggers compliance check

Compliance reporting ‚Üí CSV exports

  • S3 bucket policy required for AWS Backup write access
  • On-demand or scheduled reports
  • First-time reports can take up to 1 hour
üí° Key Insight: AWS Backup's non-destructive restore model + tag-based automation + compliance frameworks = enterprise-grade data protection with minimal manual intervention
